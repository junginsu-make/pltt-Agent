# 시나리오 및 에러 케이스

## 시나리오 A: 직원 휴가 신청 전체 흐름

이 시나리오는 프로젝트의 기준 대화입니다. 반드시 이 흐름대로 동작해야 합니다.

```
직원 A(정인수): 나 휴가 몇개 남았어?
  → [직원 LLM(라우터)] 의도: leave_inquiry → leave_agent 연결
  → [휴가 담당자 LLM] query_leave_balance(EMP-001) → {total: 15, used: 1, remaining: 14}
휴가 담당자 LLM: 15개 중 14개 남았습니다.

직원 A: 나 3월 18일에 휴가를 쓰고 싶어
  → [휴가 담당자 LLM] validate_date(EMP-001, 2026-03-18) → {valid: true, day: "수요일"}
휴가 담당자 LLM: 어떤 이유로 휴가를 쓰실 예정이신가요?

직원 A: 개인사정이야
휴가 담당자 LLM: 네 휴가 올려드릴게요~
  → [휴가 담당자 LLM] submit_leave_request(EMP-001, 2026-03-18, 1일, "개인사정")
  → DB: leave_requests INSERT (LV-2026-0001)
  → DB: approvals INSERT (APR-2026-0031, 승인자=김민준)
  → DB: leave_balances.pending_days += 1

[직원 A 상사(김민준) 대화창에 알림]
시스템: 직원 A가 휴가 신청을 하였습니다.
  → [상사 LLM] 자동 분석: 팀 일정 확인, 팀원 휴가 확인
  → ApprovalCard 표시: [승인][반려][질문하기]

직원 A 상사(김민준): [승인] 클릭
  → POST /approvals/APR-2026-0031/decide {decision: "approved"}
  → DB: leave_requests.status = 'approved'
  → DB: leave_balances.used_days += 1, pending_days -= 1
  → 캘린더 연동: 3/18 직원 A 휴가 등록
  → 알림 → 정인수: "휴가가 승인되었습니다!"
  → 알림 → 휴가 담당자(EMP-HR-001): "직원 A 3/18 휴가 승인 완료"
  → 알림 → 경영지원팀장(EMP-MGMT-LEADER): "직원 A 3/18 휴가 승인 완료"
  → 감사 로그: 휴가일, 승인일, 승인자, 사유 모두 기록
```

### 이후 다른 사람이 직원 A 일정을 물어볼 때

```
대표: 직원 A 일정에 대해서 알려줘
  → [대표 LLM(비서)] query_employee_schedule(EMP-001)
  → DB: leave_requests WHERE employee_id='EMP-001' AND status='approved'
회사 통합 LLM(대표 비서): 직원 A는 3월 18일 개인사정으로 휴가 일정이 있습니다.
```

## 시나리오 B: 대표 일정 조회 + 팀장 호출

```
대표: 직원 A 일정에 대해서 알려줘
  → [대표 LLM(비서)] query_employee_schedule(EMP-001) → {3/18 연차}
대표 LLM: 직원 A는 3월 18일 개인사정으로 휴가 일정이 있습니다.

대표: 경영지원팀장 호출해줘
  → [대표 LLM] call_person(EMP-MGMT-LEADER)
  → messaging-server가 DM 채널 생성 + 경영지원팀장에게 호출 알림
대표 LLM: 네, 경영지원팀장님께 호출 알림을 보냈습니다.

[경영지원팀장 대화창]
시스템: 대표님이 호출하셨습니다. [대화 열기]

[DM 채널 - 순수 메신저, LLM 관여 없음]
대표: 이번 달 인력 현황 어때?
경영지원팀장: (직접 타이핑) 현재 전원 정상 근무 중이고, 직원 A만 18일 휴가 예정입니다.
```

## 시나리오 C: 담당자 직접 개입

```
직원 B: 나 다음주에 3일 연속 휴가 쓸 수 있어?
  → [휴가 담당 LLM] query_leave_balance, validate_date
휴가 담당 AI: 잔여 5일이라 3일 가능합니다. 월~수(3/23~25)로 할까요?

[휴가 담당자(사람)가 [개입하기] 클릭]
  → channel.human_takeover = true
  → LLM 자동 응답 중지

휴가 담당자(사람): 잠깐, 직원 B님! 수요일에 전사 회의가 있어서 월~화(2일)로 조정하시거나, 목~금으로 변경하시는 게 좋을 것 같아요.
  → 이 메시지는 sender_type="human", [직접 응답] 뱃지 표시

직원 B: 아 그렇군요, 그럼 목금으로 할게요.
휴가 담당자(사람): 네, 목금 2일로 올려드릴게요!
  → 담당자가 직접 submit (Admin 기능 또는 대화에서 처리)

[휴가 담당자(사람)가 [AI에게 넘기기] 클릭]
  → channel.human_takeover = false
  → LLM 자동 응답 재개
```

## 에러 케이스 전체 (E-01 ~ E-18)

### 휴가 신청 단계

**E-01: 연차 0일**
- 상황: "휴가 쓰고 싶어" 했는데 잔여 0일
- LLM: "올해 연차를 모두 사용하셨어요 😢 잔여: 0일. 병가나 특별휴가가 필요하시면 말씀해주세요!"
- 시스템: 신청 차단

**E-02: 신청 일수 > 잔여**
- 상황: "3일 쓸래" 했는데 잔여 2일
- LLM: "잔여 연차가 2일이라 3일은 어려워요. 2일까지 가능한데 조정하시겠어요?"
- 시스템: 신청 차단, 잔여 이하로 수정하면 진행

**E-03: 중복 신청**
- 상황: 같은 날짜에 이미 pending/approved 있음
- LLM: "3월 18일은 이미 신청되어 있어요! (LV-2026-0001, 승인대기 중)"
- 시스템: DB UNIQUE 제약으로 차단

**E-04: 과거 날짜**
- LLM: "3월 10일은 이미 지난 날짜예요 😅 오늘 이후로 알려주세요!"

**E-05: 너무 먼 미래 (90일+)**
- LLM: "꽤 먼 미래네요. 맞으시죠?" + 공휴일이면 대안 제시

**E-06: 같은 날 팀원 다수 휴가**
- LLM: "3/18에 팀원 2명이 이미 휴가예요. 신청은 가능하지만 반려될 수 있어요."
- 시스템: team_conflicts 경고 + 팀장에 경고 전달

**E-07: 반차 오전/오후 미지정**
- LLM: "오전 반차(09~13시)와 오후 반차(14~18시) 중 어떤 걸 원하세요?"

**E-08: 여러 날 신청 (범위에 주말 포함)**
- "3/14~17" → LLM: "주말 제외하면 평일 2일이에요. 3/14(금)과 3/17(월) 신청할까요?"

### 결재 단계

**E-09: 팀장 반려**
- DB: approvals.status='rejected', leave_requests.status='rejected', pending_days 원복
- 직원에게: "팀장님이 반려했어요. 사유: [코멘트]. 날짜 변경하시겠어요?"

**E-10: 자동승인 타임아웃 (2시간 미응답)**
- scheduler가 감지 → 자동 승인 처리
- 팀장에게: "자동 승인되었습니다 (2시간 미응답)"
- 감사로그: actor=system, action=auto_approved

**E-11: 팀장이 질문 후 승인/반려**
- 팀장이 [질문하기] → 결재 상태 'reviewing', 자동승인 타이머 일시정지
- 대화 후 승인/반려 → 정상 처리

**E-12: 승인 전 직원 취소**
- 조건: status='pending'만 가능
- "아까 휴가 취소할래" → pending 건 조회 → 취소
- DB: status='cancelled', pending_days 원복

**E-13: 승인 후 직원 취소**
- LLM: "이미 승인됐어요. 취소하려면 팀장님 승인이 필요합니다. 요청 보낼까요?"
- 취소 결재 요청 생성 → 팀장 승인 후 취소 → used_days 원복

### 시스템/기술

**E-14: LLM API 실패**
- 3회 재시도 (1초, 2초, 4초)
- 실패 시: "일시적으로 응답이 어려워요 🙏 잠시 후 다시 시도해주세요"
- 감사로그 + 관리자 알림

**E-15: LLM 할루시네이션 방어**
- 방어 1: 숫자 데이터는 반드시 Tool 호출 강제 (프롬프트에 명시)
- 방어 2: Tool 결과와 LLM 응답의 숫자 비교 (post-processing)
- 방어 3: submit_leave_request 없이 "완료" 말하면 무효

**E-16: DB 연결 실패**
- "시스템 점검 중입니다"
- 메시지를 Redis 큐에 임시 저장 → DB 복구 후 재처리

**E-17: WebSocket 끊김**
- 자동 재연결 (1초, 2초, 4초... 최대 30초)
- 재연결 시 놓친 메시지 조회
- UI: "연결이 끊어졌습니다. 재연결 중..." 배너

**E-18: 동시 요청 (Race Condition)**
- DB UNIQUE 제약 + 트랜잭션
- 두 번째 요청 → unique violation → E-03으로 처리
